# Retrieval test fixtures

## `qdrant-dbsf-oracle.json`

Parity oracle for `RankFusion.DistributionBased`. Generated by
`scripts/regenerate-dbsf-oracle.py`, which **calls
`qdrant_client.hybrid.fusion.distribution_based_score_fusion` directly**
(qdrant-client pinned to 1.12.1 in `scripts/requirements.txt`). Degenerate
inputs (single-element lists, zero-variance lists) raise `ZeroDivisionError`
inside qdrant; the script catches per-list and emits the documented Strategos
0.5-convention extension — see
`src/Strategos.Ontology/Retrieval/RankFusion.DistributionBased.cs`.

**CI parity-guarded** — see `.github/workflows/ci.yml#dbsf-parity-guard`. Every
PR and every push to `main` runs `regenerate-dbsf-oracle.py --check`, which
regenerates the oracle in-memory and rejects the PR if it doesn't match the
committed JSON byte-for-byte. Regeneration is still manual (re-run after
bumping the qdrant-client pin or adding/removing fixture queries); CI just
catches silent drift.

```bash
python3 -m pip install -r scripts/requirements.txt
python3 scripts/regenerate-dbsf-oracle.py          # regenerate
python3 scripts/regenerate-dbsf-oracle.py --check  # what CI runs
```

The C# test `RankFusionDistributionBasedTests.DistributionBased_AgainstQdrantOracle_AllQueriesMatch_Within1eMinus9`
loads this JSON and asserts per-document fused scores match the oracle within
`1e-9` on non-degenerate inputs. Because the oracle is now qdrant's literal
output (not a Python reimplementation), the `1e-9` tolerance absorbs only
C# ↔ Python floating-point reordering on an identical algorithm.

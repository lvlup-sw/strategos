# Strategos 2.4.0 — Ontology Subsystem Refinements

**Date:** 2026-04-06
**Issues:** #28 (typed similarity search), #29 (lifecycle action filtering)
**Target package version:** Strategos.Ontology 2.4.0, Strategos.Ontology.Npgsql 2.4.0
**Downstream consumer:** Basileus `OntologyContextAssembler`, Basileus `ThinkStep.ResolveActionsForLifecycle`
**Status:** Approved (Approach E — spec-aligned sugar)

---

## 1. Problem Statement

PR #22 shipped the Ontology subsystem with type-safe domain modeling, pgvector similarity search, lifecycle state machines, and embedding providers. Two downstream Basileus integration points were wired up against APIs that *almost* exist but require ergonomic refinement to ship cleanly:

1. **`OntologyContextAssembler.SearchViaObjectSetsAsync`** is a stub waiting for a fluent typed-query chain over `ObjectSet<T>`. The underlying expression and provider already exist; what's missing is a chainable surface (`SimilarTo(q).WithMinRelevance(x).Take(n).ExecuteAsync()`) and a typed entry point on `IOntologyQuery`.

2. **`ThinkStep.ResolveActionsForLifecycle`** is wrapped in a `try/catch` fallback waiting for declarative `.ValidFromState()` syntax on action builders. The query method `IOntologyQuery.GetActionsForState` already exists and is correctly implemented; what's missing is the builder ergonomics for declaring lifecycle-permitted actions inline alongside `.Action(...)` instead of as separate `Transition(X, X).TriggeredByAction(name)` calls.

Both consumers ship with fallback paths that no-op until 2.4.0 lands. Bundling these into a single release lets us ship one NuGet bump and unblock both Basileus integration paths atomically.

## 2. Design Intent — What the Spec Already Says

`docs/reference/platform-architecture.md` §4.14.6 establishes the canonical lifecycle model:

**Single source of truth — the transition table.** Permissive actions (those that don't change state) are modeled as **self-loop transitions**, not as a parallel "permission" mechanism. The spec gives the canonical example at lines 1435-1437:

```csharp
lifecycle.Transition(PositionStatus.Active, PositionStatus.Active)
    .TriggeredByAction("ExecuteTrade")
    .Description("Trade modifies quantity but position remains active");
```

Line 1453 makes this explicit: *"Self-transitions (Active → Active via ExecuteTrade) are explicitly supported."*

**Coarse vs fine filter contract.** Line 1488 defines `GetActionsForState` as: *"Combines lifecycle transition constraints with action preconditions (§4.14.5): the lifecycle acts as a coarse filter (is the action structurally valid in this state?), while preconditions are a fine filter (does the specific instance satisfy the value predicates?)."*

The spec recognizes exactly two filter dimensions — structural (lifecycle transitions) and value-based (preconditions). It does **not** introduce a third "permission" dimension. Issue #29's proposed `ActionDescriptor.ValidFromStates` field would create that third dimension, which the spec deliberately avoids.

**Decision:** Issue #29's `ActionDescriptor.ValidFromStates` field is **not** added. The `.ValidFromState(X)` builder method is preserved as syntactic sugar that emits a self-loop transition into the existing `LifecycleDescriptor.Transitions` collection. The descriptor schema is unchanged. The runtime read path is unchanged.

## 3. Approach E — Spec-Aligned Sugar

The 2.4.0 work splits cleanly into two independent tracks. Both add only builder-layer ergonomics and one new query method; neither touches the descriptor schema or the read-side query service implementation.

### 3.1 Track A — Lifecycle Builder Sugar (Issue #29)

**`IActionBuilder.ValidFromState<TEnum>(TEnum state)`** — generic method on the existing non-generic `IActionBuilder`. Records the state name (via `state.ToString()`) on the action builder's internal list. No descriptor field; the value lives only on the builder.

**`ILifecycleBuilder<TEnum>.InitialState(TEnum)` and `.TerminalState(TEnum)`** — shorthand methods that internally call the existing `.State(X).Initial()` and `.State(X).Terminal()`. Pure ergonomics.

**`ILifecycleBuilder<TEnum>.Transition(TEnum from, TEnum to, string trigger)` overload** — shorthand for `.Transition(from, to).TriggeredByAction(trigger)`. Pure ergonomics.

**Build-time desugaring in `ObjectTypeBuilder.Build()`:** After the lifecycle and action descriptors are built, a projection pass walks every action builder's `ValidFromStates` list. For each `(action, state)` pair, it appends a `LifecycleTransitionDescriptor { FromState = state, ToState = state, TriggerActionName = action.Name }` to the lifecycle's transition collection. The result: the existing `OntologyQueryService.GetActionsForState` (`OntologyQueryService.cs:127-152`) — which derives valid actions by filtering `Transitions` on `FromState` — returns the right answer with **zero changes** to the read path.

**Validation:** AONT017 (`LifecycleTransitionBadState`) already fires when a transition references an undeclared state. Since `.ValidFromState()` desugars into a transition, this diagnostic catches typos for free. AONT020 (`LifecycleUnreachableState`) and AONT021 (`LifecycleDeadEndState`) are unaffected — self-loops correctly establish reachability without adding terminal-state ambiguity.

**Backward compatibility:** Trading domains that already use the existing `Transition(A, A).TriggeredByAction(X)` self-loop pattern continue to work unchanged. The new sugar is purely additive at the call site.

### 3.2 Track B — Typed Similarity Search Fluency (Issue #28)

**`SimilarObjectSet<T>` becomes an immutable fluent builder.** The current type wraps a `SimilarityExpression` and exposes only `ExecuteAsync`. Track B adds three immutable setters:

```csharp
public sealed class SimilarObjectSet<T> where T : class
{
    public SimilarObjectSet<T> WithMinRelevance(double minRelevance);
    public SimilarObjectSet<T> Take(int topK);
    public SimilarObjectSet<T> WithMetric(DistanceMetric metric);
    public Task<ScoredObjectSetResult<T>> ExecuteAsync(CancellationToken ct = default);
}
```

Each setter returns a new `SimilarObjectSet<T>` with an updated `SimilarityExpression` (constructed via `with` on the existing record). The expression itself is unchanged.

**`ObjectSet<T>.SimilarTo(string queryText)` is simplified.** The current overload (`ObjectSet.cs:79`) takes `topK`, `minRelevance`, `metric`, `embeddingPropertyName`, and `queryVector` as optional positional parameters. Track B reduces this to `SimilarTo(string queryText)` — all other knobs move to fluent setters on the returned `SimilarObjectSet<T>`. Since 2.4.0 is a minor bump and there are no published external consumers of these positional params (Basileus is the only consumer and is on a stub), this is acceptable churn.

**`IOntologyQuery.GetObjectSet<T>(string objectType) where T : class`** — new entry point. Returns a root `ObjectSet<T>` for the named ontology object type, resolved against the registered `IObjectSetProvider`. This becomes Basileus's call-site:

```csharp
var hits = await _ontologyQuery
    .GetObjectSet<SemanticDocument>(collection.ObjectType)
    .SimilarTo(query)
    .WithMinRelevance(collection.MinRelevance)
    .Take(collection.TopK)
    .ExecuteAsync(cancellationToken);
```

**No parallel generic provider.** Issue #28's text proposes `IObjectSetProvider<T>` as a separate interface. This is **rejected**: the existing non-generic `IObjectSetProvider` already routes by type internally (`PgVectorObjectSetProvider` uses `TypeMapper.GetTableName<T>()` to dispatch to per-type tables), and adding a parallel generic interface would force a parallel storage abstraction with no behavioral benefit. The non-generic provider is the storage layer; the generic `ObjectSet<T>` and `SimilarObjectSet<T>` are the typed user-facing surface. This separation is intentional and correct.

**No schema changes.** PR #22 already established per-type table layout via `TypeMapper.GetTableName<T>()`. Typed queries hit the same tables typed writes hit. No migration.

## 4. What Is *Not* Changing

This list is normative — reviewers should reject any task that touches an item below.

| Surface | Status | Reason |
|---|---|---|
| `ActionDescriptor` schema | Unchanged | Spec §4.14.6 collapses causal/permissive into transitions; no `ValidFromStates` field |
| `LifecycleDescriptor` schema | Unchanged | Build-time projection only appends to existing `Transitions` collection |
| `LifecycleStateDescriptor` schema | Unchanged | `Initial`/`Terminal` already present |
| `LifecycleTransitionDescriptor` schema | Unchanged | All required fields already present |
| `OntologyQueryService.GetActionsForState` body | Unchanged | Already spec-correct as-is |
| `IObjectSetProvider` interface shape | Unchanged | Storage abstraction stays non-generic |
| `SimilarityExpression` shape | Unchanged | All knobs already exist as constructor params |
| pgvector schema | Unchanged | Per-type table layout already supports typed queries |
| Existing Basileus `TradingOntology` | Unchanged | Self-loop transition pattern is preserved as the canonical model |

## 6. Migration & Compatibility

**Strategos consumers:** Strategos itself has no published consumers other than Basileus, which is on stubs. No migration needed.

**Basileus integration (separate PR, post-2.4.0 NuGet bump):**
- `OntologyContextAssembler.SearchViaObjectSetsAsync` replaces its stub with the fluent chain shown in §3.2
- `ThinkStep.ResolveActionsForLifecycle` deletes the `try/catch NotImplementedException` fallback wrapper and calls `_ontologyQuery.GetActionsForState` directly
- `TradingOntology.cs` may optionally migrate from `.Transition(Active, Active).TriggeredByAction("X")` to `.Action("X").ValidFromState(Active)` for clarity, but is not required to

**No breaking changes for existing API users.** The `ObjectSet<T>.SimilarTo(...)` parameter list narrows from many positional params to one — but the only consumer of that overload is internal Strategos test code, so this is captured under "internal churn."

## 7. Open Questions

1. **`IOntologyQuery.GetObjectSet<T>` error semantics for unregistered types** — throw `KeyNotFoundException`, return empty `ObjectSet<T>`, or return `null`? Decide in plan phase based on consistency with existing `IOntologyQuery` methods (which currently return empty lists rather than throwing).

2. **Should `.ValidFromState()` accept `params TEnum[]` for ergonomic batch declaration?** E.g., `.ValidFromState(Pending, Active)`. Plan phase can add this if desired; it's a trivial overload.

3. **Diagnostic message tuning** — when AONT017 fires from a `.ValidFromState()` desugar, should the error message mention the user-facing API (`ValidFromState(...)`) instead of the underlying `Transition(...)` to avoid confusion? Plan phase decision.

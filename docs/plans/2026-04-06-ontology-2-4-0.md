## Open Decisions Resolved by This Plan

The design doc ┬¦7 lists three open questions. The plan resolves them as follows:

1. **`GetObjectSet<T>` error semantics for unregistered types**: **Throw `KeyNotFoundException`**. Rationale: existing `IOntologyQuery` methods that take an `objectType` string (e.g., `GetActions`, `GetLinks`) currently return empty lists for unknown types, but `GetObjectSet<T>` is fundamentally different ŌĆö it returns a *query* object, not a result set. A query object for a non-existent type is a programming error, not an empty result. Throwing matches the EF Core `DbContext.Set<T>()` precedent which throws on unmapped types. Captured in task **B8**.

2. **`params TEnum[]` overload for `ValidFromState`**: **Deferred to post-2.4.0**. The single-state overload is sufficient for the Basileus integration. A `params` overload is a trivial additive change that can ship later without coordination.

3. **AONT017 message tuning when fired from a `.ValidFromState()` desugar**: **Deferred**. The existing AONT017 message ("Transition references state not declared in the lifecycle") is accurate even for desugared transitions. Refining the message to mention `ValidFromState` would require threading source-info through the desugaring pass, which is out of scope for sugar.

## Plan Corrections (2026-04-06, post-coverage-check)

Two issues were discovered during the plan-review coverage check that contradicted assumptions in the design doc:

**Correction 1 ŌĆö `SimilarityExpression` cloning strategy (affects B1/B2/B3).** `SimilarityExpression` is a `sealed class` (`ObjectSetExpression.cs:114-151`), not a record. The plan's use of `with` syntax in B1/B2/B3 cannot work as written. **Resolution:** add an `internal` copy constructor that chains to the primary constructor via `: this(...)`. This guarantees validation re-runs on every clone (a record with `init` setters would silently bypass `topK >= 1` and `minRelevance Ōłł [0,1]` checks when using `with`). The copy constructor uses nullable parameters for optional overrides:

```csharp
internal SimilarityExpression(
    SimilarityExpression source,
    int? topK = null,
    double? minRelevance = null,
    DistanceMetric? metric = null)
    : this(
        source.Source,
        source.QueryText,
        topK ?? source.TopK,
        minRelevance ?? source.MinRelevance,
        metric ?? source.Metric,
        source.EmbeddingPropertyName,
        source.QueryVector,
        source.Filters)
{
}
```

This preserves ┬¦4's "`SimilarityExpression` shape unchanged" guarantee literally ŌĆö the type stays a class, the public API is identical, no inheritance change. The `internal` accessor scopes the mutation surface to `SimilarObjectSet<T>` (same assembly) and prevents external callers from constructing arbitrary mutations. This is also consistent with sibling expression types in `ObjectSetExpression.cs` ŌĆö all of which are classes deriving from the abstract `ObjectSetExpression`.

**Correction 2 ŌĆö Task B9 reformulated as InMemory integration test.** The design doc and plan both claim B9 should use a `pgvector/pgvector:pg17` docker image "via the existing test container setup." Verification: `Strategos.Ontology.Npgsql.Tests.csproj` has no `Testcontainers.*` package, `src/Directory.Packages.props` has no Testcontainers entry, and all existing `PgVector*Tests.cs` files are pure SQL-string-generation unit tests using `SqlGenerator.Build*Sql(...)`. **There is no test container fixture to mirror.**

**Resolution:** B9 is reformulated as an InMemory end-to-end integration test using `InMemoryObjectSetProvider`, which already implements `ExecuteSimilarityAsync<T>` with real cosine similarity against stored embeddings (`InMemoryObjectSetProvider.cs:19-80`). What B9 is meant to verify is the *wiring* ŌĆö that the fluent chain produces a correctly-shaped `SimilarityExpression` and that `OntologyQueryService.GetObjectSet<T>` ŌåÆ `ObjectSet<T>.SimilarTo` ŌåÆ `SimilarObjectSet<T>` setters ŌåÆ `provider.ExecuteSimilarityAsync` is end-to-end correct. The `PgVectorObjectSetProvider`'s SQL execution paths are already covered by the existing `PgVectorSimilarityTests` (SQL string assertions). Adding a Docker container test for one task does not improve coverage and adds significant infrastructure (~100+ lines of fixture code, new package dep, Docker requirement) for no behavioral gain. The test moves from `Strategos.Ontology.Npgsql.Tests` to `Strategos.Ontology.Tests` since it no longer touches the Npgsql provider. Task B9's body is updated below.

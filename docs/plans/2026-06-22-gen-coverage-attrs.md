# Plan — Generators stamp `[GeneratedCode]` + `[ExcludeFromCodeCoverage]` centrally (#148)

- **Workflow:** `refactor-gen-coverage-attrs` (overhaul track)
- **Issue:** lvlup-sw/strategos#148
- **Consumer:** lvlup-sw/basileus#325 (Wave 0) — de-pollutes coverage via a `Directory.Packages.props` bump; ping on release.
- **Date:** 2026-06-22

## Problem

The two Strategos source generators (`WorkflowIncrementalGenerator`, `StateReducerIncrementalGenerator`) emit ~31 top-level types into **consumer** assemblies as `.g.cs`.
Each emitter **hand-rolls** the literal `[GeneratedCode("Strategos.Generators","1.0.0")]` immediately before its type declaration — the same string copy-pasted across ~30 sites in 11 emitters, plus the generator's Mermaid wrapper, plus a nested `SagaTimeout`.
None carry `[ExcludeFromCodeCoverage]`, so every consumer's coverage denominator is polluted by ~22k+ generated lines (basileus#325 Wave 0).

The hand-rolled duplication is itself the defect: the **tool name is wrong** (the package family is `LevelUp.Strategos.*`, not `Strategos.Generators`), the **version is a stale hard-coded `1.0.0`**, and 30 copy-paste sites mean a future emitter silently omits the stamp — which has already partially happened (the Mermaid stamp lives in the generator file, not an emitter).

## Approach (central / structural — no hand-rolling)

Per the directive ("avoid handrolling, enforce structurally/centrally"), stamping moves to a **single chokepoint** and the per-site literals are deleted.

A `GeneratedCodeStamper` parses each generated source file with Roslyn and a `CSharpSyntaxRewriter` adds **both** attributes — fully `global::`-qualified, so no `using` is required — to **every top-level** type declaration (class/record/struct/interface/enum whose parent is a namespace or the compilation unit), **idempotently** (skip if `GeneratedCode` already present), **preserving all trivia** (`// <auto-generated/>`, `#nullable enable`, xmldoc, raw-string constants).
`AddStampedSource(spc, hint, source)` becomes the **only** way the generators add sources; all 10 `AddSource` sites route through it.
Tool name + version live in one `GeneratorMetadata` resolution (version read from the generator assembly's informational version at stamp-time, build-metadata stripped, fallback `AssemblyName.Version` → `"1.0.0"`).

The class of bug is then closed structurally: any present or future emitter's top-level types are stamped by construction, and an architecture-guard test fails the build if any `spc.AddSource(` bypasses the stamper.

### Why not a shared per-site helper

Replacing 30 literals with 30 `helper(sb)` calls is *still* per-site: a future emitter can forget the call and two sources of truth (the call vs. the central rule) persist. It does not satisfy "central/structural." Rejected.

## Design Requirements

| ID | Requirement |
|----|-------------|
| **DR-1** | A single central chokepoint (`GeneratedCodeStamper.AddStampedSource`) applies the attributes; all 10 `AddSource` sites (9 Workflow + 1 StateReducer) route through it. |
| **DR-2** | Every emitted **top-level** type carries `[global::System.CodeDom.Compiler.GeneratedCode("LevelUp.Strategos", <ver>)]` **and** `[global::System.Diagnostics.CodeAnalysis.ExcludeFromCodeCoverage]`. Stamper is idempotent, trivia-preserving, top-level-only. |
| **DR-3** | Zero hand-rolled per-site `[GeneratedCode]` literals remain in emitters/generator; tool name (`LevelUp.Strategos`) and version resolve **once** from `GeneratorMetadata`. |
| **DR-4** | An architecture-guard test fails the build if any `spc.AddSource(` appears outside the stamper. |
| **DR-5** | A generator-driver test enumerates **every** emitted top-level type for a representative rich workflow + `[WorkflowState]` type and asserts both attributes are present. |
| **DR-6** | Generated code still compiles for all workflow shapes (no `CS0579` duplicate-attribute / malformed output) — behavioral compile tests; CHANGELOG + docs updated; basileus pinged on release. |

## Out of scope

- `Strategos.Ontology.Generators` (analyzer-only, emits no consumer types) and `Strategos.Contracts.Codegen` (build-time console tool; output checked into repo).
- Nested generated types' own `GeneratedCode` stamp — transitively coverage-excluded by the enclosing top-level type's `[ExcludeFromCodeCoverage]`; acceptance is top-level only.
- Consumer-side coverage de-pollution proof (basileus#325 Wave 0) — validated downstream via the package bump.
- Cutting/publishing the release itself; any change to generated runtime behavior.

## Tasks

### Task 001: `GeneratedCodeStamper` + `GeneratorMetadata` (+ unit tests)
- **Implements:** DR-1, DR-2, DR-3
- **Risk tier:** high (boundary-touching: alters generated output every consumer compiles)
- **Files:** `src/Strategos.Generators/Helpers/GeneratedCodeStamper.cs` (new), `src/Strategos.Generators/Helpers/GeneratorMetadata.cs` (new), `src/Strategos.Generators.Tests/Helpers/GeneratedCodeStamperTests.cs` (new)
- **Details:**
  - `GeneratorMetadata.ToolName = "LevelUp.Strategos"`; `ToolVersion` = `static readonly`, resolved from `typeof(GeneratorMetadata).Assembly` informational version, stripped at `+`, fallback `AssemblyName.Version` → `"1.0.0"`.
  - `GeneratedCodeStamper.Stamp(string source) -> string`: `CSharpSyntaxTree.ParseText(source, CSharpParseOptions(LanguageVersion.Latest))`; rewriter visits `ClassDeclarationSyntax`/`RecordDeclarationSyntax`/`StructDeclarationSyntax`/`InterfaceDeclarationSyntax`/`EnumDeclarationSyntax` whose `Parent` is `BaseNamespaceDeclarationSyntax` or `CompilationUnitSyntax`; adds the two attribute lists (skip if `GeneratedCode` already present); moves the node's leading trivia onto the first added attribute list so xmldoc/indent survive; returns `root.ToFullString()`.
  - `AddStampedSource(SourceProductionContext spc, string hintName, string source)`: `spc.AddSource(hintName, SourceText.From(Stamp(source), Encoding.UTF8))`.
- **Tests (adequacy-judged, test-after):** top-level class/record/struct/interface/enum each stamped with both attrs; nested type NOT stamped; `// <auto-generated/>` + `#nullable enable` + raw-string `"""` constant preserved byte-for-byte aside from inserted attrs; idempotent (second `Stamp` is a no-op); file-scoped and block namespace; multiple top-level types in one file all stamped; tool name is `LevelUp.Strategos`; a version arg exists (do **not** assert an exact version string).

## Verification (DR-6, gating, not a task)

The existing behavioral compile tests (`Strategos.Generators.Behavioral.Tests`) and the `CSharpCompilation.Emit` integration tests compile generated output — they are the safety net proving no `CS0579`/malformed output after T2+T3. Run the full `Strategos.Generators*` suite + `dotnet build` as the post-merge gate.

## Dependencies & parallelism

Strictly sequential: **T1 → T2 → T3 → T4 → T5** (one cohesive change in one project; the stamper must exist before AddSource is routed, which must exist before the literals are deleted, which must exist before the enforcement tests assert the final state). Little parallel opportunity — T1's unit tests are the only independent slice.

**Orchestration note (for plan-review):** because the tasks are tightly coupled and all touch `Strategos.Generators`, fanning them across parallel worktree subagents adds integration risk without wall-clock benefit. Recommend a **single linear delegation wave** (or orchestrator-direct implementation) over parallel fan-out.
